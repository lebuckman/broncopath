Section 1
Actor Identification

Actors represent entities that interact with the BroncoPath system — either by initiating use cases or by being depended upon for data or services. Actors are categorized as Primary (main users driving system behavior), Secondary (users with more limited interaction), or External System (non-human entities that supply or consume data).

Primary Actor
CPP Student
The core end-user of BroncoPath. CPP students initiate the majority of use cases — viewing occupancy, finding rooms, planning routes, and exploring the campus map. Approximately 27,000 enrolled students at Cal Poly Pomona constitute this actor group, with the highest-value segment being commuter students and those with back-to-back schedules.
Secondary Actor
Faculty / Staff
University employees who interact with a subset of BroncoPath's features. Faculty and staff primarily use the system to locate available meeting rooms and identify low-traffic building areas between commitments. They share the same map and room-availability interface as students but are not the primary design target.
External System Actor
CPP Schedule System  (schedule.cpp.edu)
A non-human actor representing Cal Poly Pomona's publicly available class schedule system. Updated nightly, this system provides the course schedule data — including building, room number, days, and meeting times — that BroncoPath's Python scraper ingests to calculate building occupancy predictions. The system is read-only from BroncoPath's perspective; no data is written back. This is the foundational data dependency for the entire crowd prediction engine.
Section 2
Use Cases by Actor

The following use cases are derived from the core features defined in the BroncoPath Business Plan. Each use case describes a goal-oriented interaction between an actor and the system.

CPP Student Primary
ID Use Case Description
UC-S01 View Real-Time Dashboard Student opens the app and sees an at-a-glance summary of building occupancy levels across campus at the current time. The dashboard surfaces the quietest buildings and flags high-traffic zones without requiring any input.
UC-S02 View Campus Map Student browses an interactive campus map of CPP with all major buildings rendered as color-coded markers (green / yellow / red) based on predicted occupancy. Tapping a building opens its detail view. Includes: UC-S03.
UC-S03 Check Building Occupancy Student views the predicted crowd level for a selected building at the current time — including an occupancy percentage score, a density label (Low / Moderate / High), and a room-by-room availability breakdown. This use case is included by UC-S02, UC-S04, and UC-S05.
UC-S04 Find Available Room Student searches for free classrooms, study rooms, or open spaces within a selected building. Results show room number, room type, seat count, and current status (Free / In Use / Frees at [time]). Includes: UC-S03.
UC-S05 Plan Crowd-Aware Route Student selects a starting point and a destination on campus and receives ranked route options. Each option displays estimated walk time, distance, and crowd level along the path. The recommended route prioritizes crowd avoidance over shortest distance. Includes: UC-S03.
UC-S06 Filter Buildings by Density Student applies a density filter (All / Quiet / Moderate / Busy) to the campus map view, causing buildings that do not match the selected level to visually dim. This helps students quickly identify low-traffic zones.
Faculty / Staff Secondary
ID Use Case Description
UC-F01 Check Building Occupancy Faculty or staff member views the predicted occupancy level for a building they plan to visit or schedule a meeting in. Shares the same underlying use case as UC-S03, invoked through the campus map or building search.
UC-F02 Find Available Room Faculty or staff member searches for an available meeting room or quiet workspace within a building. Identical in flow to UC-S04 — the actor distinction is in intent (scheduled meetings vs. ad-hoc study) rather than system behavior.
CPP Schedule System External System
ID Use Case Description
UC-E01 Provide Schedule Data BroncoPath's Python scraper queries schedule.cpp.edu to retrieve the current semester's course schedule data — including course name, section, building, room number, days of week, start and end times, and enrollment capacity. This data feeds the crowd prediction engine. The CPP Schedule System updates its data nightly; BroncoPath re-scrapes on a matching schedule to keep predictions current.
Note on <<include>> relationships: UC-S03 (Check Building Occupancy) is the central dependency of the system. It is included by UC-S02 (View Campus Map), UC-S04 (Find Available Room), and UC-S05 (Plan Crowd-Aware Route). This reflects the architectural reality that the occupancy prediction engine is invoked by every feature that displays crowd data. These relationships are depicted in the diagram below.
